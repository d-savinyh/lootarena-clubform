# Тесты 1-5: Качественная валидация

Детальные промпты и инструкции для первых 5 тестов. Эти тесты фокусируются на **качественном анализе**: боль, мотивация, риски, старт, UX.

---

## Тест 1: Pain Depth (The Mom Test)

### Суть
Фреймворк Роба Фитцпатрика. Люди всегда говорят "классная идея!", чтобы не обидеть. Настоящая валидация — это вопросы о **прошлом поведении**, а не о будущих намерениях.

### Как проводить

Задай себе (от лица целевого юзера) 5 вопросов:

1. **Как они решают эту проблему СЕЙЧАС?** (Excel, ручками, конкурент, никак)
2. **Сколько они ПЛАТИЛИ за решение?** ($0 = боли нет. >$0 = боль реальна)
3. **Когда последний раз сталкивались с этой проблемой?** (Вчера = горит. Месяц назад = зуд)
4. **Что произойдёт если НЕ решить проблему?** (Ничего страшного = Nice-to-have. Потеря денег = Must-have)
5. **Кто ещё из их окружения жалуется на эту проблему?** (Никто = ниша. Все = массовая боль)

### Критерии оценки

| Балл | Критерий |
|------|----------|
| 9-10 | **Hair-on-fire problem.** Люди УЖЕ платят за плохое решение. Проблема ежедневная. Убытки измеримы |
| 7-8 | Проблема реальная, решают костылями, готовы платить, но не горит |
| 5-6 | Проблема существует, но люди живут с ней нормально. Nice-to-have |
| 3-4 | Проблему замечают только если спросить. Витаминка, не болеутоляющее |
| 1-2 | Проблемы не существует. Решение ищет проблему |

### Формат вывода

```
### 1. Pain Depth (The Mom Test) — X/10

**Как решают сейчас:** [Описание текущих решений]
**Платят ли за решение:** [Да/Нет, сколько]
**Частота боли:** [Ежедневно / Еженедельно / Редко]
**Последствия нерешения:** [Конкретные последствия]
**Массовость проблемы:** [Узкая ниша / Средняя / Массовая]

**Вердикт:** [2-3 предложения итога]
**Балл: X/10** — [Почему именно столько]
```

## Тест 2: JTBD & Ego Drivers

### Суть
Люди покупают не дрель, а дырку в стене. А точнее — уважение жены за повешенную полку. Jobs-to-be-Done копает до **эмоциональной и социальной мотивации**.

### Как проводить

Для каждого сегмента пользователей выпиши:

1. **Функциональная работа** — что конкретно делает продукт? (Собрать визитку)
2. **Эмоциональная работа** — как юзер хочет ЧУВСТВОВАТЬ себя? (Крутым, продвинутым)
3. **Социальная работа** — как юзер хочет ВЫГЛЯДЕТЬ для других? (Успешный фаундер)

Затем определи:

- **Push** — что мотивирует уйти от текущего решения? (Линктри слишком простой)
- **Pull** — что притягивает к новому продукту? (Красивая визитка с верифицированным MRR)
- **Anxiety** — что пугает при переходе? (Потеряю ссылки, придётся настраивать)
- **Inertia** — что удерживает на старом решении? (Уже привык, лень переносить)

### Критерии оценки

| Балл | Критерий |
|------|----------|
| 9-10 | Продукт попадает в **статус, эго или страх** (потерять деньги/репутацию). Push сильный, Pull мощный |
| 7-8 | Чёткая эмоциональная и функциональная работа. Pull понятный |
| 5-6 | Только функциональная работа (утилита). Нет эмоционального драйвера |
| 3-4 | Работа размытая. Непонятно зачем юзеру "нанимать" этот продукт |
| 1-2 | Продукт не выполняет ни одну значимую работу |

### Формат вывода

```
### 2. JTBD & Ego Drivers — X/10

**Целевой сегмент:** [Кто]

| Тип работы | Описание |
|------------|----------|
| Функциональная | [Что делает] |
| Эмоциональная | [Как хочет чувствовать себя] |
| Социальная | [Как хочет выглядеть] |

**Силы переключения:**
- Push: [Что толкает от старого решения]
- Pull: [Что тянет к новому]
- Anxiety: [Что пугает]
- Inertia: [Что удерживает]

**Вердикт:** [2-3 предложения]
**Балл: X/10** — [Почему]
```

## Тест 3: Premortem (Red Team)

### Суть
Наш мозг оптимистичен — мы не хотим думать о плохом. Premortem заставляет представить: **проект провалился. Почему?**

### Как проводить

Перенесись на **18 месяцев вперёд**. Проект закрыт. У тебя 0 юзеров, 0 дохода. Напиши:

1. **3 фатальные причины провала** — не «мало маркетинга», а конкретно:
   - Какой именно сценарий убил проект?
   - Через какой канал пришла смерть? (конкуренты, фрод, юнит-экономика, выгорание)
   - Можно ли было предотвратить и как?

2. **2 способа обмана/фрода** — как юзеры могут абьюзить систему?

3. **1 чёрный лебедь** — неожиданное событие, которое может убить нишу

### Критерии оценки

| Балл | Критерий |
|------|----------|
| 9-10 | Риски минимальны. Каждый сценарий смерти имеет простую митигацию. Фрод не критичен |
| 7-8 | Есть 1-2 серьёзных риска, но все с проработанной митигацией |
| 5-6 | 2-3 серьёзных риска, митигация возможна но дорогая |
| 3-4 | Фатальные риски без очевидной митигации. 1+ сценарий почти неизбежен |
| 1-2 | Множество неустранимых рисков. Провал почти гарантирован |

### Формат вывода

```
### 3. Premortem (Red Team) — X/10

**Сценарий:** Февраль 2028. Проект закрыт.

**Фатальная причина №1:** [Название]
> [Подробный сценарий 2-3 предложения]
> **Митигация:** [Что сделать заранее]

**Фатальная причина №2:** [Название]
> [Подробный сценарий]
> **Митигация:** [Что сделать]

**Фатальная причина №3:** [Название]
> [Подробный сценарий]
> **Митигация:** [Что сделать]

**Фрод-сценарии:**
1. [Как юзеры обманут систему]
2. [Ещё один способ]

**Чёрный лебедь:** [Неожиданное событие]

**Вердикт:** [Итог]
**Балл: X/10** — [Почему]
```

## Тест 4: Cold Start Analysis

### Суть
Главный убийца маркетплейсов. Тестеры не придут без проектов. Фаундеры не придут без тестеров. Как разорвать круг?

### Как проводить

Определи тип продукта:

1. **Single-side** (утилита для одного юзера) → проблема холодного старта минимальна
2. **Multi-side** (маркетплейс, платформа) → критическая проблема

Для multi-side:
- Какая сторона **более ценна** (кто платит)?
- Какую сторону **проще набрать** (дешевле/быстрее)?
- Есть ли **standalone value** (ценность без второй стороны)?

Предложи **3 хака для старта** по книге Эндрю Чена «The Cold Start Problem»:
- Do things that don't scale — немасштабируемые действия
- Seed supply — засеять предложение вручную
- Anchor tenant — найти одного крупного клиента

### Критерии оценки

| Балл | Критерий |
|------|----------|
| 9-10 | Single-side продукт или multi-side с очевидным standalone value. Холодный старт не проблема |
| 7-8 | Multi-side, но одна сторона набирается легко. Есть standalone value |
| 5-6 | Multi-side, обе стороны нужны одновременно, но есть 2-3 реалистичных хака |
| 3-4 | Multi-side, обе стороны сложно набрать, standalone value слабый |
| 1-2 | Нужно 3+ стороны или критическая масса > 1000 юзеров. Без инвестиций не взлетит |

### Формат вывода

```
### 4. Cold Start Analysis — X/10

**Тип продукта:** [Single-side / Multi-side]
**Стороны:** [Перечисли стороны если multi-side]
**Standalone value:** [Есть ли ценность без второй стороны?]

**Хак для старта №1:** [Название]
> [Конкретное описание что делать]

**Хак для старта №2:** [Название]
> [Конкретное описание]

**Хак для старта №3:** [Название]
> [Конкретное описание]

**Вердикт:** [Итог]
**Балл: X/10** — [Почему]
```

## Тест 5: Friction Audit

### Суть
Проверка UX до написания кода. На каком клике юзер скажет "да ну нахер, сложно" и закроет вкладку?

### Как проводить

Представь себя **ленивым, подозрительным юзером с клиповым мышлением** (10 секунд внимания).

Пройди мысленно весь путь:
1. Откуда юзер узнал о продукте? (реклама, рекомендация, поиск)
2. Что видит на лендинге? Понятно ли за 5 секунд что делает продукт?
3. Сколько шагов до **первого "ВАУ"** момента? (Time to Value)
4. Где нужна регистрация/оплата? (Friction Point)
5. Что нужно ввести/загрузить/настроить?
6. Когда юзер получает результат?

Для каждого шага отметь:
- **🟢 Frictionless** — юзер проходит без задержки
- **🟡 Bump** — подозревает, но продолжает
- **🔴 Wall** — может уйти

### Критерии оценки

| Балл | Критерий |
|------|----------|
| 9-10 | Time to Value < 60 секунд. 0 walls. Можно попробовать без регистрации |
| 7-8 | Time to Value < 3 минуты. 1 wall (регистрация), но после неё сразу ценность |
| 5-6 | 2-3 walls. Нужна настройка. Time to Value > 5 минут |
| 3-4 | Сложный onboarding. Нужно загрузить данные, пригласить коллег, настроить интеграции |
| 1-2 | Невозможно получить ценность без длительной настройки или оплаты |

### Формат вывода

```
### 5. Friction Audit — X/10

**User Journey:**

| Шаг | Действие | Трение |
|-----|----------|--------|
| 1 | [Что делает юзер] | 🟢/🟡/🔴 [Пояснение] |
| 2 | ... | ... |
| 3 | ... | ... |

**Time to Value:** [Сколько секунд/минут до первого "ВАУ"]
**Самый опасный friction point:** [Где больше всего риск потерять юзера]
**Рекомендация:** [Как убрать/снизить трение]

**Вердикт:** [Итог]
**Балл: X/10** — [Почему]
```
