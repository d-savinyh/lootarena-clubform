# Гайд по тестированию Skills

Подробное руководство по тестированию skills на основе Anthropic Best Practices.

---

## Уровни тестирования

Skills можно тестировать на разных уровнях в зависимости от потребностей:

| Уровень | Описание | Когда использовать |
|---------|----------|-------------------|
| Manual testing | Запросы в Claude.ai/Code, наблюдение | Быстрая итерация, нет настройки |
| Scripted testing | Автоматизированные тест-кейсы в Claude Code | Повторяемая валидация |
| API testing | Оценочные suite через Skills API | Системное тестирование |

**Правило**: Выбирай уровень тестирования по масштабу skill. Внутренний skill для маленькой команды ≠ skill для тысяч enterprise пользователей.

---

## Подход: «Iterate on a single task first»

Самый эффективный метод для первой версии:

1. **Выбери один сложный use case** — не простой, а именно тот, где skill должен проявить себя
2. **Итерируй в контексте** — работай с Claude, пробуя разные подходы, пока не найдёшь выигрышный
3. **Извлеки в SKILL.md** — перенеси рабочий подход из диалога в инструкции skill
4. **Расширяй покрытие** — добавляй тест-кейсы для других сценариев

> Это использует in-context learning Claude и даёт быстрый сигнал. Широкое тестирование на старте — ловушка.

---

## Тест 1: Triggering Tests

**Цель**: Skill загружается когда нужно и не загружается когда не нужно.

### Подготовка тест-кейсов

```
## Triggering Tests: {skill-name}

Should trigger (10+ запросов):
- "Помоги настроить workspace в ProjectHub"
- "Мне нужно создать проект в ProjectHub"
- "Инициализируй проект ProjectHub для Q4 планирования"
- "Создай задачи в ProjectHub"
- "Настрой ProjectHub для команды"

Should NOT trigger (5+ запросов):
- "Какая погода в Москве?"
- "Помоги написать Python код"
- "Создай таблицу Excel"
- "Расскажи о машинном обучении"
- "Настрой базу данных PostgreSQL"
```

### Как тестировать

1. Откройте новый диалог с Claude
2. Задайте каждый запрос из списка
3. Проверьте: загрузился ли skill?
4. Запишите результаты

### Метрика

**Target**: Skill триггерится на **> 90%** релевантных запросов.

Формула: `(количество_успешных_триггеров / общее_количество_should_trigger) * 100`

### Отладка

Спроси Claude: **«Когда бы ты использовал skill [skill-name]?»**

Claude процитирует description обратно. Сравни с тем что ожидаешь — недостающие элементы нужно добавить.

## Тест 2: Functional Tests

**Цель**: Skill выполняет задачу корректно, включая edge cases.

### Формат тест-кейсов

```
Test: Создание проекта с 5 задачами
Given: Имя проекта "Q4 Planning", 5 описаний задач
When: Skill выполняет workflow
Then:
  - Проект создан в ProjectHub
  - 5 задач созданы с правильными свойствами
  - Все задачи связаны с проектом
  - 0 ошибок API

---

Test: Создание проекта с пустым именем
Given: Пустое имя проекта
When: Skill выполняет workflow
Then:
  - Ошибка обработана gracefully
  - Пользователю предложено ввести имя
  - Нет crash или необработанных исключений

---

Test: MCP сервер недоступен
Given: MCP сервер отключён
When: Skill пытается создать проект
Then:
  - Ошибка MCP обработана
  - Пользователю объяснена проблема
  - Предложены шаги по решению
```

### Категории для проверки

- ✅ **Happy path** — стандартный сценарий работает
- ✅ **Edge cases** — пустые данные, невалидный ввод
- ✅ **Error handling** — MCP ошибки, таймауты
- ✅ **Output consistency** — одинаковый запрос → одинаковый формат ответа

### Как запускать

1. Выполни каждый тест-кейс 3-5 раз
2. Сравни результаты на консистентность
3. Запиши все расхождения

## Тест 3: Performance Comparison (Baseline)

**Цель**: Доказать что skill улучшает результаты vs работа без него.

### Формат

```
## Baseline Comparison: {skill-name}

### Без skill:
- Пользователь даёт инструкции каждый раз
- 15 сообщений туда-обратно
- 3 неудачных API вызова с retry
- 12,000 токенов потрачено
- Пользователь 2 раза корректировал Claude

### Со skill:
- Автоматическое выполнение workflow
- 2 уточняющих вопроса
- 0 неудачных API вызовов
- 6,000 токенов потрачено
- 0 коррекций от пользователя
```

### Метрики для сравнения

| Метрика | Как измерить |
|---------|-------------|
| Количество сообщений | Посчитать messages в диалоге |
| Tool calls | Сравнить количество вызовов инструментов |
| Failed API calls | Мониторинг логов MCP сервера |
| Токены | Сравнить потребление (если доступно) |
| User corrections | Посчитать сколько раз пользователь перенаправлял Claude |

## Использование skill-creator для итерации

skill-creator может помочь тестировать и улучшать:

### Создание тест-кейсов

```
«Используй skill-creator чтобы создать тест-кейсы для skill [skill-name]»
```

### Ревью skill

```
«Проверь этот skill и предложи улучшения»
```

### Итерация после обнаружения проблем

```
«Используй проблемы и решения из этого чата чтобы улучшить skill [skill-name] для обработки [конкретный edge case]»
```

**Важно**: skill-creator помогает проектировать и улучшать, но **не запускает автоматические тест-suite** и не генерирует количественные результаты оценки.

## Итерация на основе результатов

### Сигналы для улучшения

| Сигнал | Тип проблемы | Действие |
|--------|-------------|----------|
| Skill не загружается | Under-triggering | Добавить триггеры в description |
| Skill загружается на всё | Over-triggering | Добавить негативные триггеры |
| Непоследовательные результаты | Execution | Улучшить инструкции |
| API ошибки | Integration | Добавить error handling |
| Пользователь корректирует | Instructions | Упростить, добавить скрипты |

### Цикл итерации

1. Выполни тесты → запиши результаты
2. Определи слабое место (trigger / functional / performance)
3. Внеси изменения в SKILL.md
4. Обнови version в metadata
5. Повтори тесты → сравни с предыдущими результатами
