# Skill Creator — Мета-skill для управления skills

Используй этот skill для создания, валидации, тестирования и улучшения других skills в проекте.

## Когда использовать этот skill

- Создание нового skill с нуля
- Валидация существующего skill перед загрузкой
- Тестирование skill (триггеры, функциональность, производительность)
- Улучшение skill, который не триггерится правильно
- Итерация skill на основе обратной связи
- Анализ и исправление ошибок в SKILL.md

## ⚠️ Критические правила

1. **Формат файла**: Файл ОБЯЗАТЕЛЬНО должен называться `SKILL.md` (регистр важен!)
2. **Имя папки**: Только kebab-case (`my-cool-skill` ✅, `My Cool Skill` ❌)
3. **YAML delimiters**: Frontmatter ОБЯЗАТЕЛЬНО начинается и заканчивается `---`
4. **Нет XML тегов**: Символы `<` и `>` запрещены в YAML frontmatter
5. **Нет README.md**: Внутри папки skill не должно быть README.md
6. **Размер SKILL.md**: Держи основной файл < 5000 слов. Детали выноси в `references/`

---

## Принципы дизайна skills

### Progressive Disclosure (3 уровня)

Skills используют трёхуровневую систему для экономии токенов:

- **Level 1** (YAML frontmatter) — ВСЕГДА в system prompt Claude. Минимум для принятия решения о загрузке skill. Это **самая важная часть** — именно по ней Claude решает, загружать ли skill.
- **Level 2** (SKILL.md body) — загружается ТОЛЬКО когда Claude считает skill релевантным. Основные инструкции и workflow.
- **Level 3** (references/, scripts/) — загружается по необходимости. Детальные доки, примеры, справочники.

**Правило**: SKILL.md < 5000 слов. Всё остальное — в `references/`.

### Composability

Skill должен работать **наряду с другими skills**. Не предполагай что ты единственный skill в сессии. Избегай конфликтующих инструкций и слишком широких триггеров.

### Portability

Skills работают одинаково в Claude.ai, Claude Code и API. Создаёшь один раз — работает везде (если окружение поддерживает зависимости skill).

## Workflow 1: Создание нового skill

### Шаг 1: Определение use cases

Перед созданием определи 2-3 конкретных сценария использования:

```
Use Case: [Название]
Trigger: Пользователь говорит "[конкретные фразы]"
Steps:
1. [Шаг 1]
2. [Шаг 2]
Result: [Ожидаемый результат]
```

**Pro Tip**: Начни с **одного сложного use case**. Итерируй в диалоге с Claude, пока skill не справится идеально. Затем извлеки выигрышный подход в SKILL.md и расширяй покрытие.

### Шаг 2: Выбор категории skill

| Категория | Описание | Пример |
|-----------|----------|--------|
| Document & Asset Creation | Создание документов, дизайнов, кода | frontend-design |
| Workflow Automation | Многошаговые процессы | n8n-workflow-dev |
| MCP Enhancement | Улучшение работы с MCP | database-management |

### Шаг 3: Создание структуры папок

```bash
mkdir -p .agent/skills/{skill-name}/scripts
mkdir -p .agent/skills/{skill-name}/references
```

### Шаг 4: Создание SKILL.md

Используй шаблон:

```yaml
---
name: {skill-name-in-kebab-case}
description: {Что делает}. Используй когда {триггеры}. Триггеры: "{фраза1}", "{фраза2}".
metadata:
  author: {Author}
  version: 1.0.0
---
```

После frontmatter добавь:
- `# {Skill Name}` — заголовок
- `## Когда использовать` — триггеры
- `## ⚠️ Критические правила` — обязательные правила
- `## Workflow` — пошаговые инструкции
- `## Примеры` — конкретные примеры
- `## Troubleshooting` — решение проблем

### Шаг 5: Валидация

Запусти скрипт валидации:

```bash
python .agent/skills/skill-creator/scripts/validate_skill.py .agent/skills/{skill-name}
```

### Шаг 6: Тестирование

Переходи к **Workflow 4: Тестирование skill**.

## Workflow 2: Валидация существующего skill

### Автоматическая проверка

```bash
python .agent/skills/skill-creator/scripts/validate_skill.py .agent/skills/{skill-name}
```

### Ручной чек-лист

#### Структура
- [ ] Папка в kebab-case
- [ ] Файл называется точно `SKILL.md`
- [ ] Нет `README.md` в папке skill
- [ ] SKILL.md < 5000 слов

#### YAML Frontmatter
- [ ] Есть `---` в начале и конце
- [ ] `name` в kebab-case без пробелов
- [ ] `description` содержит ЧТО делает skill
- [ ] `description` содержит КОГДА использовать (триггеры)
- [ ] Нет символов `<` или `>`
- [ ] `description` меньше 1024 символов

#### Инструкции
- [ ] Чёткие и actionable шаги
- [ ] Примеры с конкретным кодом/командами
- [ ] Обработка ошибок описана
- [ ] Секция `## Troubleshooting` присутствует
- [ ] Секция `## Примеры` присутствует
- [ ] Ссылки на `references/` корректны (файлы существуют)

## Workflow 3: Улучшение skill

### Skill не триггерится (undertriggering)

**Симптом**: Skill не загружается автоматически

**Диагностика**: Спроси Claude: «Когда бы ты использовал skill [skill-name]?» — Claude процитирует description. Посмотри чего не хватает.

**Решение**: Улучши description:
1. Добавь конкретные фразы-триггеры
2. Укажи типы файлов если применимо
3. Будь более специфичен о задачах
4. Добавь технические термины и ключевые слова

```yaml
# ❌ Плохо
description: Помогает с проектами.

# ✅ Хорошо
description: Управление Sprint в Linear. Используй когда пользователь просит "создать спринт", "планирование задач", "Linear tickets".
```

### Skill триггерится слишком часто (overtriggering)

**Симптом**: Skill загружается на нерелевантные запросы

**Решение**:
1. Добавь негативные триггеры
2. Сузь scope
3. Уточни домен

```yaml
# ✅ С негативными триггерами
description: Анализ данных CSV для статистического моделирования. НЕ использовать для простого просмотра данных (используй data-viz skill).
```

### Инструкции не выполняются

**Симптом**: Skill загружается, но Claude не следует инструкциям

**Решение**:
1. **Упрости инструкции** — используй bullet points и нумерованные списки
2. **Критические инструкции в начало** — с заголовком `## ⚠️ Критические правила`
3. **Убери неоднозначные формулировки**
4. **Используй скрипты для критичных проверок** — код детерминирован, языковые инструкции — нет

```yaml
# ❌ Плохо
Убедись, что всё правильно валидируется.

# ✅ Хорошо
КРИТИЧНО: Перед вызовом create_project проверь:
- Имя проекта не пустое
- Назначен хотя бы один участник
- Дата начала не в прошлом
```

**Advanced technique**: Для критических валидаций добавь скрипт в `scripts/`, который проверяет программно, вместо того чтобы полагаться на языковые инструкции.

## Workflow 4: Тестирование skill

### Подход: «Iterate on a single task first»

Самый эффективный подход для первой версии skill:
1. Выбери **один сложный use case**
2. Итерируй в диалоге с Claude, пока не добьёшься успеха
3. Извлеки выигрышный подход в SKILL.md
4. Расширяй покрытие дополнительными тест-кейсами

### Тест 1: Triggering Tests

**Цель**: Skill загружается в нужное время и не загружается в ненужное.

Создай тест-кейсы:

```
## Triggering Tests: {skill-name}

Should trigger:
- "{Очевидная фраза 1}"
- "{Очевидная фраза 2}"
- "{Перефразированный запрос}"

Should NOT trigger:
- "{Нерелевантный запрос 1}"
- "{Запрос для другого skill}"
- "{Общий вопрос}"
```

**Как тестировать**: Задай Claude каждый запрос и проверь, загрузился ли skill.

**Метрика**: Skill должен триггериться на **90%+ релевантных запросов** (проверь на 10-20 запросах).

**Отладочная команда**: Спроси Claude: «Когда бы ты использовал skill [skill-name]?»

### Тест 2: Functional Tests

**Цель**: Skill выполняет задачу корректно.

Формат тест-кейсов:

```
Test: {Название сценария}
Given: {Начальное состояние}
When: {Действие пользователя}
Then:
  - {Ожидаемый результат 1}
  - {Ожидаемый результат 2}
  - {Нет ошибок API}
```

**Что проверять**:
- Корректные результаты для стандартных случаев
- API вызовы успешны
- Error handling работает
- Edge cases обработаны (пустые данные, невалидный ввод)

### Тест 3: Performance Comparison (Baseline)

**Цель**: Доказать что skill улучшает результаты по сравнению с работой без skill.

```
## Baseline Comparison: {skill-name}

Without skill:
- Пользователь даёт инструкции каждый раз
- {X} сообщений туда-обратно
- {Y} неудачных API вызовов
- {Z} токенов потрачено

With skill:
- Автоматическое выполнение workflow
- {A} уточняющих вопросов
- {B} неудачных API вызовов
- {C} токенов потрачено
```

### Метрики успеха

| Метрика | Целевое значение | Как измерить |
|---------|-----------------|--------------|
| Trigger rate | > 90% | 10-20 тестовых запросов |
| Tool calls per workflow | < baseline | Сравнить с/без skill |
| Failed API calls | 0 per workflow | Мониторинг логов MCP |
| User corrections needed | 0 | Наблюдать за сессиями |

## Workflow 5: Итерация на основе обратной связи

Skills — живые документы. Планируй итерации по сигналам:

### Сигналы undertriggering
- Skill не загружается когда должен
- Пользователи включают его вручную
- Вопросы «а когда его использовать?»

**Действие**: Добавь больше триггеров и ключевых слов в description.

### Сигналы overtriggering
- Skill загружается на нерелевантные запросы
- Пользователи его отключают
- Путаница в назначении skill

**Действие**: Добавь негативные триггеры, сузь scope.

### Сигналы execution issues
- Непоследовательные результаты
- Ошибки API
- Пользователю приходится корректировать

**Действие**: Улучши инструкции, добавь error handling, используй скрипты для критичных проверок.

### Workflow итерации

1. Столкнулся с edge case → запиши проблему и решение
2. Вернись к skill-creator: «Используй проблемы и решения из этого чата чтобы улучшить skill [skill-name] для обработки [edge case]»
3. Обнови SKILL.md
4. Обнови version в metadata
5. Перезапусти тесты (Workflow 4)

## Паттерны skills

Подробные паттерны с примерами — в `references/skill-patterns.md`.

### Паттерн 1: Sequential Workflow

Для многошаговых процессов в определённом порядке.

### Паттерн 2: Multi-MCP Coordination

Для workflow охватывающих несколько сервисов.

### Паттерн 3: Iterative Refinement

Для задач, требующих итеративного улучшения качества.

### Паттерн 4: Context-Aware Selection

Для выбора инструмента в зависимости от контекста.

### Паттерн 5: Domain-Specific Intelligence

Для встраивания доменной экспертизы в логику skill.

Таблица выбора:

| Сценарий | Паттерн |
|----------|---------|
| Чёткая последовательность шагов | Sequential Workflow |
| Интеграция нескольких сервисов | Multi-MCP Coordination |
| Требуется высокое качество результата | Iterative Refinement |
| Разные инструменты для разных случаев | Context-Aware Selection |
| Нужна доменная экспертиза | Domain-Specific Intelligence |

## Troubleshooting

### Error: "Could not find SKILL.md"

**Причина**: Файл назван неправильно

**Решение**:
```bash
ls -la .agent/skills/{skill-name}/
# Должно быть именно SKILL.md
mv skill.md SKILL.md  # или SKILL.MD -> SKILL.md
```

### Error: "Invalid frontmatter"

**Причина**: Ошибка в YAML

**Частые ошибки**:
```yaml
# ❌ Нет разделителей
name: my-skill
description: Does things

# ❌ Незакрытые кавычки
name: my-skill
description: "Does things

# ✅ Правильно
---
name: my-skill
description: Does things
---
```

### Error: "Invalid skill name"

**Причина**: Имя содержит пробелы или заглавные буквы

```yaml
# ❌ Неправильно
name: My Cool Skill

# ✅ Правильно
name: my-cool-skill
```

### Skill не триггерится

См. **Workflow 3 → Undertriggering** и **Workflow 4 → Triggering Tests**.

### Skill триггерится на всё

См. **Workflow 3 → Overtriggering**.

### MCP соединение не работает

**Симптом**: Skill загружается, но MCP вызовы падают

**Чек-лист**:
1. MCP сервер подключён? (Settings > Extensions > [Service] → «Connected»)
2. API ключи валидны и не истекли?
3. Правильные permissions/scopes?
4. Тест MCP без skill — работает ли? («Используй [Service] MCP для получения проектов»)
5. Имена инструментов в skill совпадают с MCP? (case-sensitive!)

### Claude не следует инструкциям

**Причины и решения**:

1. **Инструкции слишком длинные** → Упрости, используй bullet points, детали вынеси в `references/`
2. **Инструкции спрятаны** → Критическое — в начало, используй `## ⚠️` заголовки
3. **Неоднозначный язык** → Будь конкретен, используй чек-листы
4. **Advanced**: Для критических проверок — скрипт в `scripts/`. Код детерминирован, язык — нет.

### Деградация на большом контексте

**Симптом**: Skill работает медленно или теряет качество

**Причины и решения**:
1. SKILL.md слишком большой → Держи < 5000 слов, детали в `references/`
2. Слишком много skills одновременно → Не более 20-50 активных skills
3. Всё загружено inline → Используй Progressive Disclosure (ссылки на references)

## Performance Notes

При создании skill рекомендуется добавить секцию для борьбы с «laziness» модели:

```markdown
## Performance Notes
- Выполняй каждый шаг тщательно
- Качество важнее скорости
- Не пропускай шаги валидации
```

**Примечание**: Добавление таких инструкций в **промпт пользователя** эффективнее, чем в SKILL.md.
