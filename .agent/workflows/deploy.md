# Деплой

Dokploy настроен на автодеплой по тегу (Trigger Type: On Tag).
При пуше тега Dokploy автоматически подхватывает код из GitHub и деплоит.

> **Первый шаг**: прочитай `.agent/toolkit.json` → секцию `project` для получения имени проекта, n8n URL, webhook-пути, `dokploy_app_id` и исключений для .dockerignore.

> **⚠️ Если `dokploy_app_id` отсутствует** — ОСТАНОВИ деплой и попроси пользователя настроить:
>
> 1. Открыть Dokploy → найти приложение
> 2. Скопировать Application ID из URL или настроек
> 3. Добавить `"dokploy_app_id": "<ID>"` в `toolkit.json` → `project`
> Без этого уведомление в Telegram проверит статус не того приложения!

## Шаги

// turbo-all

### 0. Проверь .dockerignore (автоматически)

```bash
test -f .dockerignore && grep -qE "^\.git$" .dockerignore && grep -qE "^node_modules$" .dockerignore && echo "✅ .dockerignore OK" || echo "❌ .dockerignore требует внимания"
```

Если `.dockerignore` отсутствует или не содержит минимальных исключений (`.git`, `node_modules`):

- Создай файл автоматически на основе `.gitignore` + добавь: `.git`, `.agent`, `.agents`, `.gemini`, `specs`, `*.md`, `LICENSE`
- Добавь подпроекты из `toolkit.json` → `project.dockerignore_exclude`
- Закоммить `.dockerignore` перед деплоем

### 1. Убедись что все изменения закоммичены

```bash
git pull --rebase origin main && git status
```

Если есть незакоммиченные изменения — сначала выполни `/save`.
Если pull подтянул чужой деплой (новый тег) — предупреди пользователя.

### 2. Собери changelog (ПОДРОБНО!)

#### 2.1. Посмотри коммиты с последнего тега

```bash
git log $(git describe --tags --abbrev=0 2>/dev/null || echo "--all --reverse --format=%H | head -1" | xargs git log --oneline)..HEAD --oneline --no-decorate
```

#### 2.2. Посмотри ВСЕ изменённые файлы

```bash
git diff $(git describe --tags --abbrev=0 2>/dev/null)..HEAD --stat
```

#### 2.3. Просмотри DIFF ключевых файлов

**ОБЯЗАТЕЛЬНО** просмотри `git diff <last_tag>..HEAD -- <file>` для каждого значимого файла (экраны, компоненты, хуки, стили). НЕ полагайся только на заголовки коммитов — они часто неполные!

```bash
git diff $(git describe --tags --abbrev=0)..HEAD -- "*.tsx" "*.ts" "*.css"
```

#### 2.4. Сформируй **подробный технический changelog на русском**

**Правила changelog:**

- Группируй по категориям: 🆕 Новое / 🔧 Исправления / ⚙️ Инфраструктура
- Пиши **технически**: указывай затронутые компоненты, файлы, типы данных
- Для каждого пункта — **что изменилось**, **в каком компоненте**, **почему** (если не очевидно)
- Мелочи (переименования, импорты, форматирование) можно опускать
- Цель — по changelog можно понять контекст и при необходимости откатиться

**Плохой пример:**

```
• Гео-сортировка клубов
```

**Хороший пример:**

```
• **Гео-сортировка клубов** — `ClubList`: клубы сортируются по расстоянию через `navigator.geolocation`. Добавлен `useGeoLocation` хук, компонент `DistanceBadge` на карточке клуба (1.2 км). Fallback на алфавитную сортировку при отказе в геолокации.
```

### 3. Автоматически определи тип релиза

**НЕ спрашивай пользователя** — определи сам по правилам:

- **major** (X.0.0) — ломающие изменения (breaking changes), полная переработка архитектуры, смена API
- **minor** (0.X.0) — **новые компоненты** (`[NEW]` файлы), новые экраны, новые фичи, новые webhook-и
- **patch** (0.0.X) — всё остальное: рефакторинг, баг-фиксы, правки стилей, оптимизации, обновление зависимостей

**Алгоритм:**

1. Есть ли breaking changes или удалённые публичные API? → `major`
2. Есть ли новые файлы (`[NEW]`), новые компоненты, новые экраны? → `minor`
3. Всё остальное → `patch`

Покажи changelog и выбранный тип — **без ожидания подтверждения**, сразу переходи к шагу 3.5.

### 3.5. Оформи changelog в CHANGELOG.md

В файле `CHANGELOG.md` уже может быть секция `## [Unreleased]` с накопленными изменениями от предыдущих `/save`.

**Действия:**

1. **Переименуй** `## [Unreleased]` → `## [X.Y.Z] — YYYY-MM-DD \`vX.Y.Z\``
2. **Дополни** записями из шага 2.4, если они ещё не записаны в Unreleased
3. **Добавь пустой Unreleased** заново сразу после описания:

   ```markdown
   ## [Unreleased]

   _Нет незадеплоенных изменений._

   ---
   ```

4. Убедись что формат категорий соблюдён: 🆕 Новое / 🔧 Исправления / ⚙️ Инфраструктура / Затронутые файлы
5. Стиль: технический, с указанием компонентов и файлов

### 4. Создай тег и запуши

Сформируй **краткий** changelog (5-7 строк, самые важные пункты) для тела коммита. Это будет видно в Dokploy.

```bash
npm version <type> --no-git-tag-version && VERSION=$(node -p "require('./package.json').version") && git add package.json package-lock.json CHANGELOG.md && git commit -m "release: v$VERSION" -m "<краткий changelog>" && git tag "v$VERSION" && git push && git push --tags
```

Пример коммита:

```
release: v1.2.0

• Live-загрузка клуба
• Расширенная аналитика товаров
• Упрощение настроек API
```

**ОБЯЗАТЕЛЬНО** после пуша проверь что репозиторий чистый:

```bash
git status --short
```

Если есть незакоммиченные файлы — закоммить их ДО завершения деплоя, иначе они «просочатся» в следующий релиз.

### 5. Отправь changelog в n8n (уведомление придёт после деплоя)

Прочитай `.agent/toolkit.json` → `project.n8n_url` и `project.deploy_webhook`.

n8n workflow сам дождётся завершения деплоя в Dokploy (90 сек) и отправит уведомление:

- ✅ Если деплой успешен — отправит changelog в Telegram
- ❌ Если деплой упал — отправит предупреждение об ошибке

```bash
curl -s -X POST "<project.n8n_url><project.deploy_webhook>" \
  -H "Content-Type: application/json" \
  -d "{\"message\": \"<CHANGELOG_HTML>\", \"version\": \"v<VERSION>\", \"applicationId\": \"<project.dokploy_app_id>\"}"
```

> `applicationId` — ID приложения в Dokploy. Если не передан — по умолчанию проверяет ERP.

Формат HTML сообщения:

```
🚀 <b>{project.name} v{VERSION}</b>\n\n{CHANGELOG в текстовом формате с использованием \\n для переносов}\n\n📅 {дата и время}
```

> Если `project.n8n_url` или `project.deploy_webhook` не заданы в `toolkit.json` — **спроси у пользователя** адрес n8n и webhook-путь, затем предложи сохранить их в `toolkit.json`.

### 6. Покажи итог

Покажи сводку:

- 🏷️ **Версия**: vX.Y.Z
- 📋 **Changelog**: что вошло в релиз (подробно)
- 🚀 **Dokploy**: деплой запущен автоматически
- 💬 **Telegram**: уведомление отправлено

## Что происходит автоматически

1. `npm version` — бампает версию и создаёт тег
2. `git push --tags` — пушит тег в GitHub
3. **Dokploy** получает webhook → видит новый тег → собирает и деплоит
4. **n8n webhook** → отправляет уведомление в Telegram
